# 1. 背景与胜负目标

## 1.1 故事背景

《一骑红尘：荔枝争运战》是一场双队伍、600 个结算帧限时的贡运策略对抗赛。红方和蓝方各控制一支贡运队，从岭南出发，将荔枝运抵长安兴庆宫。比赛根据是否成功交付、交付时剩余鲜度、交付时间、皇榜任务和悬赏得分计算最终胜负。

## 1.2 参赛任务

参赛选手需要编写程序，根据比赛目标和公开信息，控制主车队移动、处理皇榜任务、领取和使用资源、派出小分队，并根据结算结果调整策略。通信协议、消息格式和字段定义见《一骑红尘：荔枝争运战 通信协议.md》。

## 1.3 一局比赛如何获胜

- 只有一方退赛时，退赛方判负。
- 双方同一结算帧退赛时，本局无效且无胜方。
- 双方都未退赛时比较最终总分，总分相同时本局直接判定平局。

## 1.4 一局比赛的基本流程

一局比赛的基本路径为：

```text
读取公开地图和当前状态
指挥主车队移动和处理
领取资源、完成皇榜任务
处理障碍、设卡、攻坚和窗口争夺
进入宫宴冲刺后完成宫门验核
到达兴庆宫交付
生成最终总分与分项明细
```

基础规则：
- 开局时，每队有 100 篓好果、0 篓坏果、鲜度 100、8 名小分队人手和 4 点护卫行动点。
- 进入宫宴冲刺阶段后，S14 朱雀门才会开放宫门验核。
- 交付条件：剩余好果大于 0、鲜度大于 0
- 比赛结束条件：双方都交付、单方退赛、双方同帧退赛，或第 600 结算帧结算完成。比赛结束后，系统统一计算最终总分。最终总分由交付时间、交付时剩余好果、交付时剩余鲜度、任务分、悬赏分和惩罚分组成。
- 队伍成功交付后，不能再领取资源、完成皇榜任务或获得新的悬赏得分。交付后动作限制和违规处理见 7.4。

## 1.5 全局单位与术语

比赛按结算帧推进，不按真实时间换算。

| 术语 | 含义 |
|---|---|
| 结算帧 | 比赛推进的基本单位。每局最多 600 个结算帧 |
| 结算帧编号 | 结算帧从 1 开始编号。第 1 结算帧表示比赛开始后的第一帧 |
| 交付时间 | 队伍成功交付时的结算帧编号。数值越小，表示交付越早 |

# 2. 地图与对战信息

## 2.1 地图公开视图

地图 Excel 只作为人工阅读辅助，包含 `完整地图_选手视图` 这一张工作表。选手可以通过该工作表查看站点位置、路线边、地形图例和天气影响区；实际结算以地图配置和比赛时下发的公开状态为准。

地图公开视图展示空间关系、路线连接和天气区域。地图显示尺寸为 80 × 60。开局时，参赛程序需读取本局地图状态；每个结算帧，参赛程序读取当前公开状态。参赛程序提交 `MOVE` 时，目标节点从当前状态允许的相邻节点中选择。

正式比赛可以使用不同地图变体。比赛地图在开局时确定并下发，局内保持固定。

`完整地图_选手视图` 由以下区域组成：

| 区域 | 主要内容 | 用途 |
|---|---|---|
| 地形图例 | 官道、水路、山路、支路等地形编码和中文名称 | 读取路线类型和移动耗时差异 |
| 天气影响区图例 | `R`、`F` 等格内标记对应的区域天气；`H` 为酷暑区域展示标记 | 判断哪些区域会受暴雨、山雾影响；酷暑实际生效时按全图天气结算 |
| 站点坐标 | `S01-S15` 的站点编号、名称、类型和坐标 | 确认起点、宫门、终点及沿途站点位置 |
| 综合地图 | 地形底色、天气标记和站点标记合并后的地图 | 确认站点和路线在地图上的空间关系 |
| 路线边一览 | 路线编号、起点、终点、路线类型、路线距离、是否双向和路径编号 | 判断两站是否连通；路线距离用于计算到站所需移动量，实际到站帧数还受每帧移动量、天气和加速影响 |

参赛程序提交动作时需要使用节点编号、目标节点编号、资源类型和任务实例编号。Excel 中的中文名称用于阅读，英文枚举和编号用于程序对接。

路线距离以本局路线边清单为准，`完整地图_选手视图` 中“路线边一览”可用于人工查看。参赛程序提交动作前，先查看当前状态中的可见资源、任务、障碍和合法相邻节点。处理规则、资源效果、任务类型、任务分值和任务处理帧数固定；处理点、资源投放、领取帧数、障碍和天气区域按本局地图配置和公开状态结算。综合地图只用于确认空间关系。

公开视图读取规则：

| 读取目标 | 查看位置 | 读取方式 |
|---|---|---|
| S03 是什么站点 | `完整地图_选手视图` 的站点坐标区域 | 找 `S03`，读取站点名、类型和坐标 |
| S03 是否可到 S07 | `完整地图_选手视图` 的路线边一览 | 找起终点包含 S03、S07 的路线；若该路线不是双向，只能按起点到终点方向通行；最终以当前状态显示的合法相邻节点为准 |
| 某路线距离是多少 | `完整地图_选手视图` 的路线边一览 | 读取该路线的类型、距离和是否双向；路线边上禁止原路倒退 |
| 某区域受哪类天气影响 | `完整地图_选手视图` 的天气影响区图例和综合地图 | 看格内 `R`、`F` 等区域天气标记，再按天气规则结算；酷暑按全图天气处理 |
| 某站能否领取资源或处理任务 | 当前状态、任务列表、资源列表和对应规则 | 按系统下发的可见状态提交对应动作，不从地图图形自行推断 |

## 2.2 站点与路线

### 2.2.1 起点、宫门与终点

固定起点、宫门与终点：

| 职责 | 节点 |
|---|---|
| 起点 | S01 岭南果园 |
| 宫门 | S14 朱雀门 |
| 终点 | S15 兴庆宫 |

S15 兴庆宫不挂载设卡、攻坚、资源、任务、小分队、窗口出牌或道路障碍。未交付队伍位于 S15 时：

```text
未验核：只能等待（WAIT），或提交返回 S14 的移动
已验核，且满足交付条件：可以提交 DELIVER
已验核，但不满足交付条件：只能等待，或提交返回 S14 的移动
```

从 S15 返回 S14 是兴庆宫的特殊移动规则。返回时无视 S14 上的设卡和道路障碍，也不清除这些阻挡。

### 2.2.2 路线类型与路线距离

路线由两个规则量决定：路线距离和路线类型。选手在 `完整地图_选手视图` 的“路线边一览”中查看这两个值。

主车队在一条路线边上移动时，系统先计算这条路需要走满多少，再计算每个结算帧走多少：

| 影响项 | 含义 |
|---|---|
| 路线距离 | 这条路在地图中的长度。距离越长，到站所需移动量越大 |
| 路线耗时系数 | 每 1 点路线距离需要走多少移动量。其他条件相同时，数值越高，到站越慢 |
| 基础每帧移动量 | 没有加速时为 1000；快马、短程马和疾行令会提高这个数值 |
| 天气通行倍率 | 暴雨、山雾命中对应路线时，会降低本帧实际移动量。1000 表示不减速，数值越高，本帧走得越少 |

| 路线类型 | 路线类型字段 | 路线耗时系数（每 1 点路线距离所需移动量） | 每结算帧鲜度损耗 |
|---|---|---:|---:|
| 官道 | `ROAD` | 1380 | 0.055 |
| 水路 | `WATER` | 1250 | 0.045 |
| 山路 | `MOUNTAIN` | 1780 | 0.07 |
| 支路 | `BRANCH` | 1550 | 0.065 |

策略支路（`BRANCH`）用于转线和改变交锋位置。皇榜任务刷新仍按 ROAD / WATER / MOUNTAIN 三类路线选择目标；支路不会单独作为任务刷新分类。支路上的动作和结算机制沿用基础规则。

快马、短程马和疾行令不改变到站所需移动量，只改变基础每帧移动量。

| 状态 | 基础每帧移动量 |
|---|---:|
| 无加速 | 1000 |
| `FAST_HORSE` | 1200 |
| `SHORT_HORSE` | 1150 |
| `RUSH_SPEED` | 1300 |

移动结算使用两个量：

```text
到站所需移动量 = 向上取整（路线距离 × 路线耗时系数）
每帧移动量 = 向下取整（基础每帧移动量 × 1000 ÷ 当前天气通行倍率）
```

| 天气情况 | 当前天气通行倍率 |
|---|---:|
| 无影响天气、未命中天气、酷暑 | 1000 |
| 暴雨实际命中水路移动 | 1350 |
| 山雾实际命中山路移动 | 1100 |

实际命中的天气和加速只影响当前结算帧的每帧移动量，已经走过的移动量不影响。清障残留通行税、设卡时间税和障碍时间税按各自规则单独结算。

### 2.2.3 双向通行与合法相邻节点

主车队之间没有物理碰撞。双方可以同时停在同一节点，也可以同时位于同一路线边。普通移动是否受阻只看敌方有效设卡、道路障碍、宫门验核等移动规则；窗口争夺和处理占用只影响对应的处理或出牌动作。

路线读取规则：

| Excel 中看到 | 代表什么 | 参赛程序怎么用 |
|---|---|---|
| 路线边一览中有 `E03`，起点 S03，终点 S07，路线类型为 `ROAD`，路线距离为 54 | S03 与 S07 之间存在一条官道，距离按 54 计算 | 只有当前状态显示 S07 是合法相邻节点时，才提交 `MOVE` 到 S07 |
| “是否双向” | `true` 表示双向；`false` 表示只能由起点到终点通行 | 提交 `MOVE` 前以当前合法相邻节点为准 |

### 2.3.4 障碍点

道路障碍按站点挂载，比赛开局时生成，中途不会随机新增。普通清障、小分队清障或 T04 完成后，会清除目标障碍。目标站点存在道路障碍时，普通移动（`MOVE`）的合法目标不包含该站；队伍可以清障、处理 T04、派小分队清障、强制通行或改走其他路线。道路障碍阻挡进入目标站点；队伍已经停在该站点时，离站是否受阻按本次目标节点判断。

| 看到的信息 | 代表什么 | 可提交动作 |
|---|---|---|
| 当前状态显示某站存在道路障碍 | 道路障碍正在该站生效 | `CLEAR`、处理对应 T04 清障任务、`SQUAD_CLEAR` 或 `FORCED_PASS` |

T04 表示任务模板编号。本局内具体清障任务实例使用任务列表中的任务实例编号提交。

道路障碍候选点示例：

| 挂载站点 | 默认障碍类型 | 相邻站点 | 规则影响 |
|---|---|---|---|

| S08 秦岭栈道 | 落石（`ROCKFALL`） | S06、S07、S09、S10 | 阻挡山路中段；影响武关与转线路线 |
| S10 武关 | 关口拥堵（`PASS_CROWD`） | S08、S09、S11 | 阻挡关键关隘；可作为 T11 / T14 目标 |
| S11 潼关驿 | 关口拥堵（`PASS_CROWD`） | S10、S12 | 阻挡终段入关路线 |

## 2.4 天气区域与地图变体

规则固定项：

| 项目 | 固定规则 |
|---|---|
| 核心角色节点 | S01(5,50) 为起点、S14(76,18) 为宫门、S15(78,18) 为终点 |
| 节点编号集合 | 正式地图使用 S01-S15，不新增节点；站点编号、编码、站点名、类型和图标固定 |
| 路线类型枚举 | 只使用 `ROAD / WATER / MOUNTAIN / BRANCH` |
| 路线方向 | 提交移动前以本局合法相邻节点为准；单向路线只能按路线方向通行 |
| 处理机制 | 前段交接、登船处理、水路换运、入关交接、宫前交接、宫门验核等机制固定 |
| 资源机制 | 冰鉴、马类、船权、过所、官凭、情报等资源效果固定 |
| 任务规则 | 任务类型、任务分值和任务处理帧数固定 |
| 天气规则 | 天气类型和天气生成规则固定 |

地图可变项：

| 项目 | 规则 |
|---|---|
| 路线边清单 | 路线数量、端点、路线类型、路线距离和是否双向以本局地图为准 |
| 路线距离 | 不同地图里，路线距离可以不同；以开局下发的为准 |

## 3.1 主车队

每队可控制、可携带、可消耗的对象包括主车队、随车货物、随车资源、小分队和护卫行动点。每队只有一支主车队；货物和资源都跟随主车队；小分队与护卫行动点是独立战术资产。

| 层级 | 内容 | 主要用途 |
|---|---|---|
| 主行动单位 | 主车队 | 移动、处理、领取资源、完成任务、清障、设卡、攻坚、验核、交付 |
| 随车货物 | 好果、坏果、鲜度、冻结好果 | 决定交付质量，也可作为部分动作成本 |
| 随车资源 | 冰鉴、马、船权、过所、官凭、情报 | 加速、保鲜、码头资源标记、窗口出牌、探路 |
| 远程战术资产 | 小分队人手 | 探路、远程清障、增援己方设卡、削弱敌方设卡 |
| 窗口战术资产 | 护卫行动点 | 支付窗口牌 `BING_ZHENG` |

主车队状态包括：

| 中文状态 | 程序状态 | 含义 |
|---|---|---|
| 空闲 | `IDLE` | 可提交合法主车队动作 |
| 移动中 | `MOVING` | 正在沿路线前进 |
| 主动等待 | `WAITING` | 主动等待（WAIT）后停在路线边上，已走移动量不增不减 |
| 处理中 | `PROCESSING` | 正在处理站点、资源、任务或清障 |

## 3.2 运送货物

初始货物：

| 项目 | 初始值 | 规则意义 |
|---|---:|---|
| 好果 | 100 篓 | 计入交付，也可支付设卡、攻坚、献贡、终局急策等成本 |
| 坏果 | 0 篓 | 不计入好果分，可用于攻坚 |
| 鲜度 | 100 | 计入鲜度品质分 |
| 冻结好果 | 0 | 主车队处理流程已冻结但尚未最终扣除的好果 |

### 3.2.1 好果和坏果

鲜度第一次低于以下任一数值时，触发好果转坏果：

```text
90、80、70、60、50、40、30、20、10
```

每首次低于一个数值，1 篓好果转为 1 篓坏果。等于阈值时不触发转坏；扣除后低于阈值才触发。若同一结算帧扣鲜度后首次低于多个尚未触发的数值，按这些数值各触发 1 次。

本节的好果总量指可支配好果和冻结好果之和。坏果只记录数量，不记录鲜度。每队同一时刻最多只有一个主车队处理流程会冻结好果；冻结好果全部归属于当前主车队处理流程。

好果转坏和鲜度归零按下表顺序处理：

| 顺序 | 情况 | 规则 |
|---:|---|---|
| 1 | 触发好果转坏 | 先转可支配好果；可支配好果不足时，再转冻结好果 |
| 2 | 好果转坏使用冻结好果 | 转坏照常生效；正在使用这部分冻结好果的处理失败，不产生收益；已损失好果不返还 |
| 3 | 好果总量变为 0 | 鲜度记为 0，坏果保留 |
| 4 | 本帧鲜度降为 0 后仍有好果 | 剩余好果全部报废，不增加坏果 |
| 5 | 鲜度归零报废使用冻结好果 | 报废照常生效；正在使用这部分冻结好果的处理失败，不产生收益；已损失好果不返还 |

### 3.2.2 鲜度

鲜度每个结算帧结束时扣除：本帧鲜度扣除值 = 每帧基础鲜度扣除值 × 天气鲜度系数 × 急策鲜度系数。天气未实际命中当前状态时，天气鲜度系数按 ×1 处理。鲜度保留实际数值；好果转坏果和交付条件都按扣除后的实际鲜度判断。

| 当前状态 | 每帧基础鲜度扣除值 |
|---|---:|
| 停靠、主动等待、处理、验核、窗口争夺、休整、强制通行额外等待 | 0.05 |
| 官道移动 | 0.055 |
| 水路移动 | 0.045 |
| 山路移动 | 0.07 |
| 支路移动 | 0.065 |
| 强制通行中实际走路线边 | 按对应路线类型取值 |

强制通行期间，因阻挡产生的额外等待按 0.05 扣鲜度；未到达目标前实际走路线边时，按对应路线类型扣鲜度。强制通行倒计时结束并到达目标节点的这一帧，按停靠、等待等空闲状态扣鲜度，每帧基础鲜度扣除值为 0.05。

| 修正来源 | 鲜度系数 |
|---|---:|
| 酷暑（HOT） | ×1.5 |
| 暴雨区域内移动、登船处理、水路换运 | ×1.3 |
| 山雾（MOUNTAIN_FOG） | ×1 |
| 疾行令（RUSH_SPEED）生效中 | ×1.25 |
| 护果令（RUSH_PROTECT）生效中 | ×0.2 |

到站所在结算帧仍按本帧实际完成的路线类型扣鲜度，不因到站后路线类型清空而改按停靠扣除。

鲜度结算后限制在 0-100。冻结好果被清算不单独扣鲜度。

### 3.3.1 保鲜资源

冰鉴只提高当前鲜度，不重置已经触发过的好果转坏果记录，也不会把坏果转回好果。冰鉴在本结算帧鲜度扣除前生效。
鲜度已为 0 时，冰鉴使用会被拒绝。

### 3.3.2 移动资源

快马（`FAST_HORSE`）与短程马（`SHORT_HORSE`）不叠加；新的马类效果会覆盖旧的马类效果。马类资源使用成功后，本结算帧立即按新的基础每帧移动量结算；使用当帧不扣减持续效果剩余帧数。后续结算帧开始时按 7.1 的持续效果规则扣减。疾行令（`RUSH_SPEED`）不覆盖马类效果，二者冲突。已有马类效果时使用疾行令会失败，且不消耗终局急策次数和好果；疾行令生效中使用快马或短程马会失败，且不消耗对应马类资源。

### 3.3.3 文书资源

# 4. 基础动作规则

本章规定每帧动作上限、动作参数、移动处理、资源领取与使用、验核交付和失败处理。

## 4.1 每帧动作上限

每个结算帧按动作类别限制提交数量：

| 类别 | 上限 |
|---|---:|
| 主车队动作 | 1 |
| 小分队动作 | 1 |
| 窗口出牌动作 | 1 |
| 终局急策动作 | 1 |

同类动作超过上限时，该类别全部非法；该类别本帧不执行任一动作。每个超过上限的动作类别按 1 次同类动作冲突记录，不按该类别内提交的动作条数重复计数。

```text
主车队类别冲突：主车队本结算帧按系统等待（WAIT）处理，并记录同类动作冲突
小分队类别冲突：本结算帧不执行小分队动作，并记录同类动作冲突
窗口出牌类别冲突：本次窗口出牌按弃权（ABSTAIN）处理，并记录同类动作冲突
终局急策类别冲突：本结算帧不执行终局急策动作，并记录同类动作冲突
```

同一结算帧内，不同类别动作如果共享同一种资源，按 7.1 的每帧结算顺序逐步校验、预占和扣除。先结算的动作先占用资源；后结算的动作若资源不足，按该动作的成本不足规则失败。终局急策动作类别用于限制疾行令、护果令和破关令；疾行令和护果令作为主车队动作提交，因此同时受主车队动作上限限制。绑定破关令（`BREAK_ORDER`）虽然跟随 `BREAK_GUARD` 或 `VERIFY_GATE` 提交，也占用终局急策动作额度。

窗口出牌动作只有本队正在参与窗口，且当前结算帧是出牌拍时才有效。

主车队动作：

| 中文动作 | 程序动作 | 用途 |
|---|---|---|
| 等待 | `WAIT` | 主动等待；移动中会暂停本结算帧前进 |
| 移动 | `MOVE` | 前往合法相邻节点 |
| 通用处理 | `PROCESS` | 处理当前站点配置的处理点 |
| 领取资源 | `CLAIM_RESOURCE` | 在资源点领取资源 |
| 使用资源 | `USE_RESOURCE` | 使用已持有资源 |
| 处理皇榜任务 | `CLAIM_TASK` | 完成指定任务并获得任务分 |
| 主车队清障 | `CLEAR` | 完成 6 个结算帧处理后清除道路障碍，不获得 T04 分 |
| 设卡 | `SET_GUARD` | 在当前节点建立己方设卡 |
| 攻坚破卡 | `BREAK_GUARD` | 攻击敌方设卡 |
| 强制通行 | `FORCED_PASS` | 支付时间税通过设卡或障碍 |
| 宫门验核 | `VERIFY_GATE` | 宫宴冲刺后在宫门完成交付前验核 |
| 交付 | `DELIVER` | 在终点完成最终交付 |
| 码头登船 | `DOCK` | 登船处理可提交 `DOCK`，也可提交 `PROCESS` |

小分队动作：小分队探路（`SQUAD_SCOUT`）、小分队清障（`SQUAD_CLEAR`）、小分队增援（`SQUAD_REINFORCE`）、小分队削弱（`SQUAD_WEAKEN`）。

参赛程序按动作含义选择合法动作。站点处理类型由系统根据当前位置和公开状态判断；动作的具体填写方式不在本任务书展开。

`PROCESS` 和 `DOCK` 的目标必须是当前节点；目标不是当前节点时，动作会被拒绝。

主线基础动作包括 `MOVE`、`WAIT`、`PROCESS`、`VERIFY_GATE`、`DELIVER`。

## 4.2 等待与移动

移动必须满足：

```text
目标节点是当前合法相邻节点
当前节点到目标节点存在本局路线边；若路线为单向，必须符合起点到终点方向
当前节点没有未完成的必经处理
目标没有阻止本次通行的敌方设卡或道路障碍
```

队伍位于 S15 兴庆宫且需要返回 S14 朱雀门时，按 2.2.1 的特殊移动规则处理。

普通 `MOVE` 的目标由本局路线边相邻端点推导，再按必经处理、敌方设卡和道路障碍过滤。资源、任务或公开处理对象正在被占用，不会阻止主车队用普通 `MOVE` 进入该节点；到达后能否处理对应对象，按对象忙、窗口争夺或处理规则另行判断。目标存在敌方设卡或道路障碍时，按 6.3 的攻坚破卡或强制通行规则处理。

`WAIT` 表示主动等待；移动中提交 `WAIT` 会暂停本结算帧路线前进。主车队停在节点上时提交 `WAIT`，本帧只等待，不进入路线边，下一结算帧仍按所在节点可提交动作。单帧未提交动作时，系统执行等待；系统等待不会主动刹车，移动中的主车队可继续沿原路线前进。

移动中等待和继续前进按下表处理：

| 当前情况 | 本帧提交 | 结果 |
|---|---|---|
| 路线边移动中 | 未提交动作或提交空动作 | 继续向当前目标节点前进 |
| 路线边移动中 | `WAIT` | 本帧暂停前进，路线进度不增加 |
| 已因 `WAIT` 悬停在路线边上 | 未提交动作或提交空动作 | 恢复向原目标节点前进 |
| 已因 `WAIT` 悬停在路线边上 | `WAIT` | 继续悬停 |
| 路线边移动中或悬停中 | 提交前往当前目标节点的移动 | 继续向当前目标节点前进 |

主车队已经在路线边上时，本段路线起点表示当前路线边的起点，当前目标节点表示本段路线终点。此时 `MOVE` 的合法目标为：当前目标节点，或本段路线起点的其他合法相邻节点。提交 `MOVE` 到其他合法相邻节点时，队伍放弃旧进度，从本段路线起点改道出发，新路线从 0 开始累计。路线边上不能提交返回本段路线起点的移动来原路倒退。

到站当帧只完成位置更新；目标节点上的任务、验核、交付、处理或清障，从下一结算帧开始提交和结算。

休整中（RESTING）只允许等待（WAIT）或不提交动作。其他动作不生效，不延长、不重置休整，本帧休整倒计时正常减少。

## 4.3 站点处理

参赛程序按当前位置提交对应处理动作。系统根据当前位置匹配处理类型。

处理规则：

```text
处理类型由当前站点决定
普通处理点提交 PROCESS
S04 登船处理可以提交 PROCESS 或 DOCK
DOCK 只在当前节点处理类型为 BOARD 时有效
S14 宫门验核提交 VERIFY_GATE
处理机制固定；处理点挂载站点、处理类型和处理帧数以本局状态为准
处理被中断后进度清零，不产生收益
```

## 4.4 领取资源与使用资源

资源领取动作需要指定目标节点和资源类型。

资源效果固定；资源投放站点、数量和领取帧数以本局地图配置和公开库存为准。参赛程序按系统下发的可见资源状态判断某站是否可领取。

资源领取规则：

| 场景 | 系统反馈 | 处理结果 |
|---|---|---|
| 主车队停靠在目标资源节点，且目标节点等于当前节点 | 可提交 | 可以提交领取资源（`CLAIM_RESOURCE`） |
| 领取成功 | 领取成功（`RESOURCE_CLAIM`） | 目标节点该资源库存 -1，本队该资源 +1 |
| 位置不符 | 动作被拒绝（`ACTION_REJECTED`）；原因：不在目标节点（`NOT_AT_TARGET_NODE`） | 不进入处理，不扣资源，不计非法动作 |
| 库存不足、资源类型缺失或目标节点没有该资源 | 动作被拒绝（`ACTION_REJECTED`）；原因：资源不足（`RESOURCE_NOT_ENOUGH`） | 不进入处理，不扣资源，不计非法动作 |
| 处理完成时库存不足 | 动作被拒绝（`ACTION_REJECTED`）；原因：资源不足（`RESOURCE_NOT_ENOUGH`） | 本次领取失败，不获得资源 |
| 同一队伍领取不同资源类型 | 分别提交 | 每种资源类型分别提交领取动作；领取开始和处理完成时都要求该站该资源仍有库存 |
| 背包上限 | 无额外上限 | 持有数量由地图库存、领取结果和使用消耗决定 |

资源使用动作需要指定资源类型；需要目标节点的资源，按对应资源规则指定目标。

`USE_RESOURCE` 为瞬发主车队动作。主车队停靠在节点时，可以提交 `USE_RESOURCE`。路线边移动中或路线边主动等待时，只能提交 `WAIT`、`MOVE`、`RUSH_SPEED`，或使用马类资源；提交冰鉴、情报、过所、官凭等非马类资源，会被拒绝，不消耗资源。主车队正在处理站点、验核或强制通行时，提交 `USE_RESOURCE` 不生效、不扣资源、不计非法动作，当前处理继续推进；休整中提交 `USE_RESOURCE` 会被拒绝，不扣资源、不计非法动作，休整倒计时继续。提交后不改变当前目标节点。马类资源使用成功时，本帧仍按当前方向结算路线移动，并影响本帧移动量。`ICE_BOX`、马类资源和 `INTEL` 的效果见第 3 章资源表。

鲜度已为 0 时，冰鉴使用会被拒绝。

## 4.5 宫门验核与终点交付

提交宫门验核必须同时满足：

| 提交要求 | 规则结果                                                     |
| -------- | ------------------------------------------------------------ |
| 比赛阶段 | 只能在宫宴冲刺阶段提交；普通阶段提交会被拒绝，不进入处理，不创建宫门窗口，不消耗破关令或探路标记 |
| 队伍位置 | 主车队位于 S14                                               |
| 验核状态 | 队伍尚未完成验核                                             |
| 队伍状态 | 不在移动、窗口争夺、休整、强制通行、通用处理或宫门验核中     |

提交交付（`DELIVER`）时，必须同时满足下表全部交付条件。只满足其中一部分时，不会完成交付。

| 交付条件 | 不满足时系统反馈 | 处理结果 | 是否计非法动作 |
|---|---|---|---|
| 主车队位于 S15 兴庆宫 | 动作被拒绝；原因：未到终点（`DELIVER_NOT_AT_TERMINAL`） | 不进入交付结算 | 否 |
| 已完成宫门验核，且已验核状态已经生效 | 动作被拒绝；原因：未验核（`DELIVER_NOT_VERIFIED`） | 不进入交付结算 | 否 |
| 尚未交付 | 动作被拒绝；原因：已交付（`ALREADY_DELIVERED`） | 不重复结算，不触发交付后违规惩罚 | 否 |
| 剩余好果大于 0 | 动作被拒绝；原因：交付条件不满足（`DELIVER_REQUIREMENT_NOT_MET`） | 不进入交付结算 | 否 |
| 鲜度大于 0 | 动作被拒绝；原因：交付条件不满足（`DELIVER_REQUIREMENT_NOT_MET`） | 不进入交付结算 | 否 |

如果队伍处于下列状态，本帧不能执行交付：

| 提交时队伍状态 | 系统反馈 | 处理结果 | 是否计非法动作 |
|---|---|---|---|
| 正在移动中 | 动作被拒绝；原因：移动中禁止动作（`MOVING_ACTION_FORBIDDEN`） | 不进入交付结算 | 否 |
| 正在休整中 | 动作被拒绝；原因：休整中禁止动作（`RESTING_ACTION_FORBIDDEN`） | 不进入交付结算，休整继续推进 | 否 |
| 正在处理、强制通行或宫门验核中 | 不执行本次交付动作 | 原处理继续推进，不进入交付结算 | 否 |
| 正在窗口争夺中 | 本帧按窗口争夺处理 | 本次主车队动作不执行，不进入交付结算 | 否 |

# 5. 皇榜任务与窗口争夺

## 5.1 皇榜任务规则

皇榜任务必须由主车队提交处理皇榜任务动作完成，并指定本局任务实例编号。

任务模板编号表示任务类型，本局任务实例编号表示本局内唯一任务实例。同一种任务模板可以多次刷新成不同任务实例；同一任务实例完成后，后续重复处理该任务实例会被拒绝。

| T04 | 清障任务 | 主车队在目标障碍节点或相邻节点完成 `CLAIM_TASK` 处理，并同步清障 | 30 | 6 |
| T06 | 争马换乘 | 主车队位于目标节点；启动任务时消耗 1 个快马（`FAST_HORSE`）或短程马（`SHORT_HORSE`），并完成 `CLAIM_TASK` 处理 | 30 | 3 |
| T08 | 码头争船 | 主车队位于目标节点并完成 `CLAIM_TASK` 处理 | 30 | 4 |

| T14 | 山口急递 | 主车队位于目标节点并完成 `CLAIM_TASK` 处理 | 15 | 5 |

T04 目标丢失规则：

```text
普通 CLEAR 或小分队清障只清除障碍，不获得 T04 分
T04 处于可处理状态时，目标障碍被普通清障、小分队清障、系统事件或其他非 T04 任务方式清除，T04 进入目标丢失失败
T04 目标丢失后不重新刷新替代任务
```

## 5.4 窗口争夺

窗口争夺处理双方同一结算帧争夺同一公共权益；资源领取还支持后手争抢窗口。公共权益包括同一份资源、同一任务处理权、同一固定处理站点的处理权、宫门验核权、障碍处理权和通行窗口权益。窗口固定 3 拍。

窗口创建于第 T 帧，第 T+1 帧为第 1 拍出牌，第 T+2 帧为第 2 拍出牌，第 T+3 帧为第 3 拍出牌并结算窗口结果。

### 5.4.1 窗口触发条件

| 对象 / 场景 | 是否生成窗口 | 处理结果 | 是否扣非法动作分 |
|---|---|---|---|
| 同一任务、同一资源、同一固定处理站点、宫门验核，或主车队同帧争抢同一障碍处理 | 是 | 生成对应窗口争夺，权益由窗口结果决定 | 否 |
| 资源领取中被另一方第 1 次打断 | 是 | 原领取流程中断，资源库存进入窗口保护；窗口胜方从头开始领取 | 否 |
| 同一资源第 2 次被打断 | 否 | 返回对象忙 | 否 |
| 已开始处理的非资源对象被后续再抢 | 否 | 后提交方返回对象忙；原处理继续推进 | 否 |
| 同一空节点同帧设卡（SET_GUARD） | 否 | 系统按公平账本（具体规则不公开）选出一方开始设卡处理，另一方返回对象忙 | 否 |
| 同一障碍上 T04 与 CLEAR 同帧提交 | 是 | 按同一障碍争夺处理；窗口胜方继续自己的原动作 | 否 |
| 强制通行遇敌方有效设卡 | 是 | 创建通行窗口（PASS），守方为设卡归属方；守方因自己的设卡被强制通行而被动参战 | 否 |
| 仅强制通行道路障碍 | 否 | 不创建 PASS 窗口；按本队本次强制通行处理，不清除障碍 | 否 |
| 破关悬赏 | 否 | 随攻坚破卡结算是否完成悬赏 | 否 |

参赛程序不能按红蓝、队伍编号、提交顺序或到达顺序推断先手。表内明确写为对象忙的场景，以表格处理结果为准。

| 窗口类型 | 触发条件 |
|---|---|
| `RESOURCE` | 双方争抢同一节点同一资源；或一方正在领取资源时，另一方后续到同一节点领取同一资源 |
| `TASK` | 双方争抢同一任务实例的处理权 |
| `GATE` | 双方争抢宫门验核 |
| `DOCK` | 双方争抢同一固定处理站点；适用于前段交接、登船处理、水路换运、入关交接和宫前交接；不含宫门验核 |
| `PASS` | 攻方提交 `FORCED_PASS`，守方为目标节点有效设卡归属方；守方因自己的设卡被强制通行而被动参战 |
| `OBSTACLE` | 双方争抢同一障碍处理 |

通行窗口（`PASS`）的守方参战原因是自己的有效设卡被强制通行，这是被动参战，不要求守方本帧也提交动作或停在目标节点。守方原本的移动、处理、验核、强制通行或休整继续结算。守方若处于处理中、验核中、强制通行中、休整中或已退赛，本次通行窗口每拍只能按弃权（`ABSTAIN`）处理；即使提交本窗口出牌，也不生效、不扣成本、不计非法动作。

已交付队伍因交付前留下的设卡被动进入通行窗口时，系统自动按弃权处理，不扣交付后违规分。已交付队伍主动提交任何 `WINDOW_CARD`，动作不生效，并按 7.4 的交付后违规动作处理。

同一队伍在同一结算帧同时参与多个窗口时，只能提交 1 个窗口出牌动作。该出牌只对匹配的窗口编号生效；其他窗口本次出牌按弃权（`ABSTAIN`）处理。

### 5.4.2 窗口出牌

窗口出牌必须指定窗口编号和窗口牌。

出牌拍允许提交窗口出牌的主状态包括空闲、等待中、移动中和窗口争夺中。未提交窗口出牌、提交空动作，或提交的窗口编号不匹配本队当前参与窗口时，对应窗口本次出牌按弃权（`ABSTAIN`）处理。窗口出牌缺少窗口编号或窗口牌时，该动作按参数非法处理，对应窗口本拍没有有效出牌，按弃权结算；提交未知窗口牌或同一帧提交多张窗口牌，仍按非法动作或同类动作冲突处理。

### 5.4.3 窗口牌与出牌成本

参赛程序只提交窗口牌，不指定成本来源。系统按下表规则自动扣除即时成本；出牌提交时不提前扣成本，亮牌结算时才校验并即时扣除。

| 牌 | 成本 |
|---|---|
| `YAN_DIE` | 消耗 1 个可用文书资源；`PASS_TOKEN` 和 `OFFICIAL_PERMIT` 都可支付验牒成本，二者没有消耗优先级差异 |
| `QIANG_XING` | 本队已有马类移动增益或疾行令（`RUSH_SPEED`）正在生效时，本拍免消耗；否则依次消耗 1 个 `FAST_HORSE`、1 个 `SHORT_HORSE` |
| `XIAN_GONG` | 鲜度至少 80，且消耗 1 篓可支配好果，不能使用冻结好果 |
| `BING_ZHENG` | 消耗 1 点护卫行动点 |
| `ABSTAIN` | 无成本 |

成本不足时，该牌失效，本次窗口出牌按弃权（ABSTAIN）处理。窗口牌成本一旦在亮牌结算时扣除，就不返还；已扣好果不参与本结算帧末尾的好果转坏果清算。弃权不克制任何牌；一方出有效牌、另一方弃权时，有效牌方获得本拍胜点，双方都弃权时本拍为平点。

### 5.4.4 窗口胜负与平局

克制关系：

| 我方 \ 对方 | `YAN_DIE` | `QIANG_XING` | `XIAN_GONG` | `BING_ZHENG` |
|---|---|---|---|---|
| `YAN_DIE` | 平 | 胜 | 负 | 负 |
| `QIANG_XING` | 负 | 平 | 胜 | 负 |
| `XIAN_GONG` | 胜 | 负 | 平 | 胜 |
| `BING_ZHENG` | 胜 | 胜 | 负 | 平 |

每拍胜方获得 1 拍胜点。3 拍结束后，胜点高者获得窗口权益；胜点相同为 `DRAW`。

窗口平局速查：

| 规则点 | 处理 |
|---|---|
| 对象归属 | 平局没有胜方，双方不共享对象权益；资源、任务、固定处理站点、宫门或障碍处理都不结算给任一方 |
| 休整 | 非通行窗口双方主车队进入休整 3 个结算帧；通行窗口（PASS）只让攻方强制通行注销并休整 3 个结算帧，守方不休整，也不暂停原动作 |
| 是否计入重复平局 | 计入。同一连续争夺周期内，同一对象最多允许 2 次平局窗口 |
| 冷却期间可做动作 | 对象进入平局冷却；冷却期间提交该对象动作会被拒绝，且不算非法动作、不扣分、不消耗资源、不启动处理；队伍可以改做其他合法动作 |

重复平局抑制按同一争夺对象计算：

| 争夺对象 | 连续平局上限 | 达到上限后的处理 |
|---|---:|---|
| 任务、资源、普通固定处理站点、障碍、通行窗口 | 2 次 | 该对象进入 18 个结算帧平局冷却 |
| 宫门验核（`GATE`） | 2 次 | 宫门对象冷却 6 个结算帧；冷却期间再次提交宫门验核，不创建窗口，也不启动宫门验核处理 |

窗口产生胜方、对象被处理、对象过期，或平局冷却结束后，连续争夺周期重置。

对象进入平局冷却后，会在窗口或对象状态中显示为 `SUPPRESSED`。任务即使仍显示为 `ACTIVE`，该对象处于 `SUPPRESSED` 时，处理或争夺动作会被拒绝。

窗口在第 T+3 结算帧判定平局时，`RESTING` 状态会在第 T+3 结算帧写入；第 T+3 结算帧不扣休整倒计时，休整倒计时从下一结算帧开始扣。`SUPPRESSED` 的截止帧包含在冷却期内；截止帧当帧仍按平局冷却处理，下一结算帧起按冷却结束后的状态重新判断。

平局冷却结束后的下一结算帧，该对象进入新的连续争夺周期；上一周期的平局次数不再沿用。

### 5.4.5 窗口结束后的处理

窗口结果处理：

| 场景 | 结算 |
|---|---|
| 窗口产生胜方 | 窗口结束后的下一结算帧，系统沿用胜方原动作开始处理或继续处理 |
| 资源领取打断窗口产生胜方 | 胜方从头开始领取资源；原领取方不恢复旧进度 |
| 窗口期间任务到达原截至帧 | 任务倒计时暂停；窗口结束前不会自然过期 |
| 胜方获得任务处理权，但任务已到达原截至帧 | 胜方获得 1 次胜方保护处理机会；保护处理成功则得分，失败或离开合法站位则进入保护期中断后过期 |
| 窗口平局 | 双方都不获得保护处理机会；本次争夺动作注销 |
| 任务窗口平局且未达平局上限 | 双方 `RESTING` 的 3 个结算帧内任务有效期暂停；若任务未到截至帧，休整后恢复为可争夺；若平局时已到截至帧，任务过期 |
| 任务进入平局冷却 | 本次 `DRAW` 触发的双方 `RESTING` 3 个结算帧和后续冷却期间，任务有效期都暂停；冷却结束后按剩余有效帧数继续计算 |
| 资源窗口平局 | 资源留在原节点库存中 |
| 通行窗口平局 | 攻方本次强制通行注销并休整 3 个结算帧；守方保持原状态 |

任务窗口结束后，胜方只获得任务处理权；是否得分取决于后续任务处理是否成功完成。败方不获得任务处理权和任务分；后续再抢同一任务，按对象忙或任务保护中处理。

例：双方同帧争抢同一任务实例后进入 `TASK` 窗口。胜方不再重新提交处理皇榜任务动作，系统从窗口结束后的下一结算帧沿用原动作开始任务处理；任务处理完成后才获得任务分。

# 6. 拦截、突破与终局急策

## 6.1 道路障碍

### 6.1.1 障碍如何阻挡

道路障碍：

| 障碍类型 | 程序字段 | 可挂载或影响的站点 |
|---|---|---|
| 落石 | `ROCKFALL` | 山路沿线站点、栈道站点、关隘附近站点 |
| 水涨 / 洪水 | `FLOOD` | 水路沿线站点、码头、水驿 |
| 泥泞 | `MUD` | 官道、山路、策略支路沿线站点 |
| 码头阻塞 | `DOCK_BLOCK` | 码头站点、登船站点、水路换运站点 |
| 关口拥堵 | `PASS_CROWD` | 普通关隘、关键关隘、入关交接站点 |
| 断桥 | `BROKEN_BRIDGE` | 含桥梁语义的站点 |
| 山体滑坡 | `LANDSLIDE` | 山路入口、山道、栈道等山路沿线站点 |

| 处理方式 | 动作 | 结果 |
|---|---|---|
| 主车队清障 | `CLEAR` | 完成 6 个结算帧处理后清除道路障碍，不获得 T04 分 |
| 任务清障 | 处理对应 T04 清障任务 | 得到 T04 分，并同步清除障碍 |
| 小分队清障 | `SQUAD_CLEAR` | 延迟清除障碍，不获得 T04 分 |
| 强制通行 | `FORCED_PASS` | 本队本次通过，不清除障碍 |


### 6.1.2 清障与强制通行

主车队清障（`CLEAR`）提交时必须指定目标节点。当前节点必须等于目标障碍节点，或与目标障碍节点直接相连。若从相邻节点完成 `CLEAR`，只清除目标障碍，不自动进入目标节点；提交时目标不存在道路障碍会被拒绝，处理完成前目标被清除时，本次清障不产生结果。

清障残留规则：

```text
主车队清障或小分队清障成功后，系统记录清障方和清障结算帧
清障方后续通过该障碍节点，不支付残留通行税
若清障在第 X 结算帧完成，非清障方在第 X 到第 X+30 结算帧提交进入原障碍节点的普通 MOVE 时，额外支付 6 个结算帧残留通行税；第 X+31 结算帧起提交时不再支付
开始通过按提交本次普通 MOVE 的结算帧判断
残留通行税在 MOVE 开始时直接加入本次路线总耗时
残留通行税期间按移动中处理，目标站点动作会被拒绝
残留通行税不恢复障碍，也不改变 T04 已完成或已失败的状态
T04 清障任务完成后不创建清障残留
强制通行不清除障碍，也不创建清障残留
```

## 6.2 设卡

### 6.2.1 如何设卡

设卡目标必须是主车队当前所在节点，注意 S15 兴庆宫禁止设卡。道路障碍不算设卡，有道路障碍的站点仍可主动设卡。

每个节点同一时刻最多存在 1 个有效设卡；目标节点已有有效设卡时，再次新增设卡动作合法但不生效。双方同一结算帧在同一空节点设卡时，不生成窗口；系统侧按公平账本选出（具体规则不公开）一方开始设卡处理，另一方按业务拒绝处理。设卡处理完成后，才会生成设卡。

| 主动设卡位置 | 是否允许设卡 | 基础成本 | 防守值上限 |
|---|---|---:|---:|
| 普通可设卡站点 | 允许 | 0 篓好果 | 6 |
| 关键关隘 | 允许 | 1 篓好果 | 7 |
| 宫门 S14 朱雀门 | 允许 | 1 篓好果 | 4 |
| 已有道路障碍的站点 | 允许 | 0 篓好果 | 5 |
| 终点 S15 兴庆宫 | 禁止 | - | - |

设卡处理帧数为 4 个结算帧。提交时可以额外投入 0、1 或 2 篓好果；额外投入小于 0 或大于 2 时，动作被拒绝，不进入设卡处理。
设卡完成后，防守值 = `min(节点防守值上限, 2 + 额外投入好果 × 2)`。超过节点防守值上限的投入不会继续提高防守值，已投入成本不返还。

每队最多同时保留 2 个有效设卡节点。新设卡完成后，若本队有效设卡数量超过 2 个，系统移除本队最早完成的有效设卡；被移除设卡已扣除的成本不返还。

设卡完成当帧不拦截敌方；从下一结算帧开始阻挡敌方。

### 6.2.2 防守值与风化

设卡会随帧数增加产生风化效果，每次风化效果会导致设卡防守值减少 1 点，最低降至 0。

| 主动设卡位置 | 首次风化 | 后续风化间隔 |
|---|---|---|
| 普通可设卡站点 | 设卡完成后 30 个结算帧 | 每 30 个结算帧 |
| 关键关隘，且完成时防守值 ≥ 4 | 设卡完成后 45 个结算帧 | 每 30 个结算帧 |
| 关键关隘，且完成时防守值 < 4 | 设卡完成后 30 个结算帧 | 每 30 个结算帧 |
| 宫门 S14 朱雀门 | 设卡完成后 30 个结算帧 | 每 30 个结算帧 |
| 已有道路障碍的站点 | 设卡完成后 30 个结算帧 | 每 30 个结算帧 |

防守值降至 0 时该设卡失效并移除：不再阻挡通行，不再支持计入该设卡的关悬赏触发次数，不能作为攻坚、小分队增援或削弱的目标，也不能被后续增援重新激活。

## 6.3 突破敌方阻挡

### 6.3.1 攻坚破卡

攻坚破卡动作需要指定目标节点、投入好果或坏果；进入宫宴冲刺后，可以按终局急策规则绑定破关令。

单次攻坚投入的好果和坏果各最多 2 篓。目标必须是相邻节点，并且存在敌方有效设卡；防守值已经为 0 的设卡无法作为攻坚目标。攻坚破卡只处理设卡本身；清除相邻节点设卡后，主车队仍停在原节点，之后还要提交移动才能进入目标节点。

提交时声明投入必须有足够可支配好果和坏果；不足时动作被拒绝，不扣资源、不改变设卡、不进入休整。攻坚进入结算后，声明投入的好果、坏果正式扣除；无论成功、失败、只削减防守值或攻坚值为 0，已扣成本都不返还。

攻坚破卡是本帧结算的主车队动作，没有额外处理帧数。攻坚结算完成后，若没有进入休整，下一结算帧按当前位置和目标状态提交后续动作；若攻坚失败进入休整，休整期间只允许等待或空动作。

攻坚强度由投入物决定：

| 投入 | 攻坚效果 |
|---|---|
| 好果 | 每 1 篓提供 2 点攻坚值 |
| 坏果 | 每 1 篓提供 3 点攻坚值 |
| 终局急策 `BREAK_ORDER` | 本次攻坚额外 +3 攻坚值 |

| 条件 | 结果 |
|---|---|
| 攻坚值不低于目标防守值 | 攻坚成功，防守值清零 |
| 攻坚值低于目标防守值，但大于 0 | 攻坚失败，防守值按本次攻坚值减少，攻方 `RESTING` 5 个结算帧 |
| 攻坚值为 0，且目标仍有防守值 | 攻坚不生效 |


### 6.3.2 强制通行

目标节点存在敌方有效设卡或道路障碍，强行进入目标节点即为强制通行（`FORCED_PASS`）动作。强制通行目标必须是当前节点可到达的相邻节点，并且目标上必须有敌方有效设卡或道路障碍。目标无阻挡时，强制通行会被拒绝且不启动。

强制通行只让本队本次通过，不清除设卡或道路障碍。

| 目标阻挡 | 处理 |
|---|---|
| 只有道路障碍 | 不创建通行窗口，直接开始强制通行 |
| 有敌方有效设卡 | 先创建通行窗口（`PASS`）；窗口触发和守方被动参战规则见 5.4 |

通行窗口结算：

| 窗口结果 | 处理 |
|---|---|
| 攻方胜出 | 窗口结束后的下一结算帧开始强制通行 |
| 守方胜出 | 本次强制通行失败；攻方留在发起节点，休整 `min(8, max(3, ceil(通行窗口创建时时间税 × 0.25)))` 个结算帧；休整结束后可重新选择动作 |
| 平局 | 本次强制通行取消；攻方休整 3 个结算帧；休整结束后可重新选择动作 |

强制通行总需求帧数 = 路线耗时 + 时间税。路线耗时按 2.2.2 的移动规则计算，时间税按下表计算。计算完成后，后续天气、马类或速度变化不重新计算本次总需求帧数。

| 阻挡来源 | 基础时间税 | 时间税上限 | 计算规则 |
|---|---:|---:|---|
| 普通可设卡站点设卡 | 10 个结算帧 | 40 个结算帧 | `min(40, 10 + 当前防守值 × 5)` |
| 关键关隘设卡 | 15 个结算帧 | 50 个结算帧 | `min(50, 15 + 当前防守值 × 5)` |
| 宫门设卡 | 12 个结算帧 | 32 个结算帧 | `min(32, 12 + 当前防守值 × 5)` |
| 已有道路障碍的站点设卡 | 8 个结算帧 | 28 个结算帧 | `min(28, 8 + 当前防守值 × 5)` |
| 目标节点道路障碍 | 8 个结算帧 | 28 个结算帧 | 8 个结算帧 |

同时存在主动设卡和道路障碍时，分别计算时间税，取较高者。
目标只有道路障碍时，只按强制通行开始结算帧时有效的道路障碍计算时间税；目标有敌方有效设卡时，只按通行窗口创建结算帧时有效的设卡和道路障碍计算时间税。之后新增的设卡或道路障碍不计入本次强制通行时时间税计算。


强制通行期间提交其他主车队动作不生效、不扣成本、不计非法动作，强制通行继续推进。

强制通行鲜度扣减：

| 帧类型 | 鲜度扣减 |
|---|---|
| 额外等待帧 | 按基础值 0.05 |
| 路线移动帧 | 按当前路线类型扣 |
| 到达目标节点当帧 | 按基础值 0.05 |

强制通行重复限制：

| 情况 | 规则 |
|---|---|
| 本队完成强制通行并到达目标节点 | 该目标节点成为本队上一次强制通行到达节点 |
| 主车队停在上一次强制通行到达节点 | 不能再次提交强制通行；提交后动作被拒绝，原因：强制通行重复（`FORCED_PASS_REPEAT`） |
| 主车队离开后又回到该节点 | 仍按上一次强制通行到达节点处理，不能在该节点再次提交强制通行 |
| 本队之后完成新的强制通行 | 上一次强制通行到达节点更新为新的目标节点 |
| 强制通行失败，未到达目标节点 | 不更新上一次强制通行到达节点 |

### 6.3.3 破关悬赏

破关悬赏：

| 类型 | 适用节点 | 分值 |
|---|---|---:|
| 普通悬赏（`NORMAL_BOUNTY`） | 普通可设卡节点 | 10 |
| 关键关隘悬赏（`KEY_BOUNTY`） | 关键关隘节点 | 18 |

| 触发情况                                                     | 生成结果         |
| ------------------------------------------------------------ | ---------------- |
| 某支队伍在该节点设卡后，连续 30 个结算帧没有被清除，且第 30 个结算帧结束时防守值仍 > 0 | 生成破关悬赏     |
| 某支队伍在该节点设卡后，连续 60 个结算帧没有被清除，且这 60 个结算帧内防守值始终 > 0 | 生成破关悬赏     |
| 某支队伍在该节点设卡后，敌方累计 2 次有效进攻失败            | 生成破关悬赏     |
| 某支队伍在关键关隘设卡后，敌方对该关隘累计完成 3 次合法攻坚破卡结算，或累计 3 次因强制通行触发通行窗口 | 生成关键关隘悬赏 |

有效进攻失败计数：

| 情况                                                         | 计数结果 |
| ------------------------------------------------------------ | -------- |
| 敌方对可设卡节点上的有效设卡发起攻坚破卡，攻坚值大于 0 且未攻破 | +1       |
| 敌方对可设卡节点上的有效设卡发起强制通行，守方胜出或平局导致敌方未通过 | +1       |
| 敌方成功攻破可设卡节点上的设卡                               | 不计     |
| 敌方攻坚值为 0，设卡防守值不变                               | 不计     |
| 敌方成功强制通行通过可设卡节点上的设卡                       | 不计     |
| 普通移动（MOVE）被设卡挡住                                   | 不计     |

关键关隘攻防计数（keyPassCombatCount）：

| 情况                                                         | 计数结果                                |
| ------------------------------------------------------------ | --------------------------------------- |
| 对关键关隘有效设卡完成一次合法攻坚破卡结算                   | +1，攻坚成功、攻坚失败、攻坚值为 0 都计 |
| 对关键关隘有效设卡发起合法强制通行，并创建通行窗口           | +1，窗口胜负不影响计数                  |
| 非法动作、平局冷却拒绝、目标防守值为 0、目标不是关键关隘有效设卡 | 不计                                    |

悬赏完成与得分限制：

| 判断步骤 | 规则 |
|---|---|
| 悬赏存在 | 目标节点存在活跃破关悬赏，或本次攻坚先触发并创建悬赏 |
| 攻坚结果 | 攻坚前目标防守值大于 0，且本次攻坚值足以清零防守值时，设卡被清除；否则不发悬赏分 |
| 得分条件 | 本次攻坚破卡所在结算帧开始时，攻破方公开总分低于设卡方公开总分，才获得悬赏分 |
| 未满足得分条件 | 只清除设卡，不获得悬赏分；主车队仍停在原位置 |

关键关隘第 3 次合法攻坚可以同时触发悬赏和结算悬赏。若本次攻坚累计到第 3 次，并且本次攻坚又打破该设卡，系统会先生成关键关隘悬赏，再立即结算该悬赏。是否加悬赏分，按本帧开始时攻破方是否落后判断。

## 6.4 小分队行动

小分队通用规则：

```text
所有小分队动作提交时都必须指定目标节点编号
提交时目标需要满足对应战术条件；到达生效帧会再次按同一条件校验
动作通过校验后立即消耗人手，并进入在途状态
到达生效结算帧后重新校验目标状态；目标无效或落地失败时，人手不返还
在途小分队落地不占本帧新提交小分队动作名额；某帧有小分队落地时，本帧仍可新提交 1 个小分队动作
```

单次战术规则：

| 战术                          | 消耗人手 | 到达时必须满足         | 成功结果                                                     |
| ----------------------------- | -------: | ---------------------- | ------------------------------------------------------------ |
| 小分队探路（SQUAD_SCOUT）     |        1 | 目标节点存在           | 目标节点获得本队探路标记                                     |
| 小分队清障（SQUAD_CLEAR）     |        2 | 目标仍为道路障碍       | 目标道路障碍被清除，本队成为清障方。对方 30 个结算帧内通过该节点，加 6 个结算帧残留通行税 |
| 小分队增援（SQUAD_REINFORCE） |        2 | 目标为本队有效设卡节点 | 目标本队有效设卡防守值 +2，不超过该节点防守值上限            |
| 小分队削弱（SQUAD_WEAKEN）    |        2 | 目标为敌方有效设卡节点 | 目标敌方有效设卡防守值 -2。不足 2 时降为 0，设卡不再阻挡通行 |

增援、削弱只作用于防守值大于 0 的有效设卡。小分队提交成功时就按下表确定几帧后生效，派出小分队不会移动主车队。

计算方式：

| 项目     | 规则                                                         |
| -------- | ------------------------------------------------------------ |
| 可派范围 | 不设额外距离上限。目标节点满足对应战术条件即可提交           |
| 起算位置 | 停靠时用当前节点坐标，移动中用本段路线起点坐标               |
| 目标位置 | 目标节点坐标                                                 |
| 距离 D   | 切比雪夫距离公式：横向差 `dx = abs(x1 - x2)`，纵向差 `dy = abs(y1 - y2)`，`D = max(dx, dy)` |
| 基础延迟 | min(15, max(3, ceil(D / 3)))。距离超过 45 时仍按 15 帧计算   |
| 山雾影响 | 提交时目标节点处于山雾作用区域，且战术是小分队探路（SQUAD_SCOUT）时，基础延迟 +2 帧，仍不超过 15 帧 |
| 其他天气 | 暴雨、酷暑和晴朗不影响小分队生效帧                           |
| 生效帧   | 提交帧 + 最终延迟。提交成功后，后续天气变化不重算            |

### 6.4.1 探路

| 动作 | 消耗人手 | 效果 |
|---|---:|---|
| 小分队探路（`SQUAD_SCOUT`） | 1 | 对目标节点添加己方探路标记，后续可让适用动作处理帧数减少 3 个结算帧，最低处理帧数为 2 |

探路标记按生成帧计算可用期。第 X 结算帧生成的探路标记，在第 X 到第 X+45 结算帧都可用；第 X+46 结算帧开始时被清理。

同一动作只消耗 1 个探路标记。多个同队同节点探路标记不叠加减时，成功完成适用动作时消耗最早获得的 1 个；动作失败、中断或窗口平局时不消耗探路标记。

探路标记适用于领取资源、处理皇榜任务、宫门验核、主车队清障和通用处理。探路标记是公开节点状态；对方能看到标记归属，本队标记仅归属队伍可用。

### 6.4.2 远程清障

| 动作 | 消耗人手 | 效果 |
|---|---:|---|
| 小分队清障（`SQUAD_CLEAR`） | 2 | 延迟清除目标道路障碍，不获得 T04 分 |

小分队清障提交时只派出人手，不占用目标障碍，也不会立刻触发障碍窗口。到达生效帧目标仍为道路障碍才成功；目标已消失或不再是障碍时失败，已消耗人手不返还。

### 6.4.3 增援与削弱设卡

| 动作 | 消耗人手 | 效果 |
|---|---:|---|
| 小分队增援（`SQUAD_REINFORCE`） | 2 | 己方有效设卡防守值 +2，不超过该节点防守上限 |
| 小分队削弱（`SQUAD_WEAKEN`） | 2 | 敌方有效设卡防守值 -2，最低降到 0 |

增援和削弱只作用于防守值大于 0 的有效设卡。目标为非对应归属的有效设卡，或到达时防守值已降为 0，本次小分队动作失败，已消耗人手不返还。

## 6.5 宫宴冲刺与终局急策

宫宴冲刺触发：

```text
第 390 结算帧前不会触发宫宴冲刺
第 450 结算帧时，若对局尚未结束，必定进入宫宴冲刺
第 390-449 结算帧内，任一主车队已经到达 S14 朱雀门，进入宫宴冲刺
第 390-449 结算帧内，任一主车队不在终段交锋节点，且到 S14 的最短路线距离不超过 15，进入宫宴冲刺
第 390-449 结算帧内，任一主车队不在终段交锋节点，且按基础最快路线估算 60 个结算帧内可到达兴庆宫，进入宫宴冲刺
```

这里的“最短路线距离”按路线边距离求和。接近判定使用基础最快路线估算，估算项排除坐标格数、天气、马类、疾行令、干扰、设卡、障碍、窗口争夺、处理流程和任务处理。终段交锋节点固定为 S11、S12、S13；“接近宫门”和“60 个结算帧内到终点”判定使用 S11、S12、S13 之外的节点，第 450 结算帧仍强制触发。正常到达 S14 时仍会因“到达 S14”条件触发。

参赛程序提交动作时，按当前比赛阶段和动作返回结果判断是否已经进入宫宴冲刺。基础最快路线估算由系统完成，参赛程序不需要计算该公式。

宫宴冲刺触发后：

```text
比赛进入宫宴冲刺阶段
开放 S14 宫门验核；宫门验核只标记已验核状态，不发皇榜任务分
开放终局急策
禁止新提交小分队动作
已在途小分队继续按原到达帧落地
仍可在窗口中提交 BING_ZHENG，前提是护卫行动点足够
宫宴冲刺持续到双方都交付、退赛结束或第 600 结算帧上限
```

宫宴冲刺只改变比赛阶段、宫门开放和终局急策开放，不会给已验核或已交付队伍回溯发放皇榜任务分。

终局急策：

| 急策 | 成本 | 持续帧数 | 效果 |
|---|---|---:|---|
| `RUSH_SPEED` | 2 篓好果 | 15 | 基础每帧移动量变为 1300，鲜度损耗系数 ×1.25 |
| `RUSH_PROTECT` | 不消耗果品 | 30 | 鲜度损耗系数 ×0.2 |
| `BREAK_ORDER` | 优先消耗 2 篓坏果；坏果不足 2 篓且好果至少 1 篓时消耗 1 篓好果 | 本次绑定动作 | 攻坚值 +3，或验核处理帧数 -3，最低 3 个结算帧 |

# 7. 结算与比赛规格

## 7.1 比赛结束

系统按固定顺序结算每帧动作，并在比赛结束时计分、扣分。

交付（`DELIVER`）必须同时满足以下条件：

```text
主车队位于 S15 兴庆宫
已经完成宫门验核，且已验核状态生效
剩余好果大于 0
鲜度大于 0
队伍不处于移动、窗口争夺、休整、强制通行、通用处理或宫门验核中
```

## 7.2 得分来源

最终总分由六类正向得分减去惩罚得到。

任务相关数值会影响送达基础分、用时分和皇榜任务分：

| 名称 | 规则 |
|---|---|
| 普通任务基础分累计 | 只统计普通皇榜任务；T01/T02/T04/T06/T08/T11 每个 30 分，T12/T13/T14 每个 15 分 |
| 任务参与折扣 | 普通任务基础分累计低于 90 时，送达基础分和用时分会被折减；达到 90 后不再折减 |
| 用时任务系数 | min(1, 普通任务基础分累计 / 90) |
| 任务里程碑奖励 | 普通任务基础分累计达到 60 / 90 / 110 时，皇榜任务分额外 +15 / +20 / +15 |

普通任务基础分累计本身不作为最终得分项，只影响送达基础分、用时分和皇榜任务分。

| 分项 | 上限 | 计分规则 |
|---|---:|---|
| 皇榜任务分 | 180 | 普通任务基础分累计 + 任务里程碑奖励，封顶 180 |
| 送达基础分 | 240 | 完成终点交付才有；普通任务基础分累计不足 90 时折减，0 分任务也有 50% 保底 |
| 用时分 | 69 | 先按交付时间算原始用时分，再乘用时任务系数；当前公式下最高为 69 |
| 好果数量分 | 180 | 交付时剩余好果越多越高 |
| 鲜度品质分 | 180 | 交付时鲜度越高越高 |
| 破关悬赏分 | 100 | 完成破关悬赏获得；基础悬赏最高 80，完成任意悬赏后额外 20 |
| 惩罚项 | 扣分 | 非法动作、交付后违规动作 |

普通设卡、普通攻坚、普通窗口争夺、普通资源获取和普通资源使用不直接给分。

已交付队伍的主要分项公式如下。所有取整均向下取整。

| 分项 | 公式 / 规则 |
|---|---|
| 送达基础分 | `min(240, 120 + floor(普通任务基础分累计 × 4 / 3))` |
| 好果数量分 | `floor(交付时剩余好果 / 100 × 180)` |
| 鲜度品质分 | `floor(交付时剩余鲜度 / 100 × 180)` |
| 原始用时分 | `floor((600 - 交付时间) / 600 × 70)` |
| 用时分 | `floor(原始用时分 × min(普通任务基础分累计, 90) / 90)` |
| 皇榜任务分 | `min(180, 普通任务基础分累计 + 里程碑奖励)`；里程碑奖励为小于 60 加 0，60-89 加 15，90-109 加 35，110 及以上加 50 |
| 破关悬赏分 | 若破关悬赏原始分大于 0，则为 `min(破关悬赏原始分, 80) + 20`；否则为 0 |

## 7.3 未交付结算

未交付限制：

| 情况 | 结算限制 |
|---|---|
| 未完成交付 | 送达基础分、好果数量分、鲜度品质分、用时分均为 0 |
| 未完成交付但完成了皇榜任务 | 皇榜任务分最多只计 80，不应用任务里程碑奖励 |
| 未完成交付但完成了破关悬赏 | 破关悬赏分最多只计 25，不应用悬赏完成额外奖励 |

未交付队伍按未交付限制表结算：已完成的皇榜任务和破关悬赏保留对应上限。若双方最终总分相同，包括双方都是 0 分，本局仍按总分平局处理。

未交付时，皇榜任务分 = `min(普通任务基础分累计, 80)`；破关悬赏分 = `min(破关悬赏原始分, 25)`。

最终总分：

所有正向分相加，再扣除惩罚；最低计为 0 分。

## 7.4 惩罚、退赛与失联

| 惩罚类型                               | 适用阶段                                                     | 扣分规则                            | 上限 |
| -------------------------------------- | ------------------------------------------------------------ | ----------------------------------- | ---: |
| 普通非法动作惩罚                       | 队伍未交付前，或动作不属于交付后违规规则时                   | 前 5 次不扣分；第 6 次起每次扣 1 分 |   20 |
| 交付后违规惩罚                         | 队伍已交付后，提交等待、空动作心跳、系统等待、重复交付以外的动作 | 每次扣 5 分                         |   30 |

失联与退赛：

```text
失联与退赛只适用于未交付队伍；已交付队伍不再累计失联退赛
每个结算帧，参赛程序需要在本帧规定截止前返回动作
未交付队伍连续 10 个结算帧未收到动作：记录失联警告
未交付队伍连续 60 个结算帧未收到动作：记录退赛
```

## 8.5 常见拒绝与非法动作速查表

| 类别 | 常见情形 | 规则结果 |
|---|---|---|
| 普通非法动作 | 不存在或当前阶段禁止的动作；必需目标缺失；公开参数格式非法；移动到非相邻节点；同类动作超上限 | 计入普通非法动作次数；前 5 次不扣分，第 6 次起每次扣 1 分，最多扣 20 分 |
| 业务拒绝 | 对象忙；队伍状态不满足；资源不足或不可用；公开目标不可处理；重复完成；对象处于平局冷却；任务处于胜方保护期 | 动作不生效，不扣分，不增加普通非法动作次数 |
| 迟到或错帧 | 动作对应结算帧不一致，或超过本帧接收截止才到达 | 不进入本帧动作列表，不计入普通非法动作次数 |
| 已交付后违规 | 已交付后提交等待、空动作、系统等待和重复交付以外的动作 | 每次扣 5 分，最多扣 30 分；不再叠加普通非法动作扣分 |
